# AI-Sprint 프로젝트 기획서

## 1. 프로젝트 개요

AI-Sprint는 창고관리자 및 재고관리자를 위한 실시간 재고관리 웹 애플리케이션입니다. 사용자는 재고 목록을 군별로 확인하고, 특정 품목을 선택해 품목 사진, 규격, 위치, 수량, 안전재고, 입출고 상태, 최근 변경 이력까지 한 화면에서 파악할 수 있습니다.

이 프로젝트의 1차 목표는 실제 WMS 또는 ERP 전체 기능을 모두 구현하는 것이 아니라, 재고 현황 파악과 품목 상세 조회에 집중한 MVP를 설계하는 것입니다.

## 2. 대상 사용자

### 창고관리자

- 창고 내 품목 위치와 수량을 확인한다.
- 부족 재고와 입출고 대기 상태를 빠르게 확인한다.
- 품목의 실제 보관 위치와 관리 기준을 확인한다.

### 재고관리자

- 품목별 현재고, 안전재고, 최소 수량을 관리한다.
- 군별 재고 현황과 부족 재고를 분석한다.
- 재고 변경 이력을 확인하고 이상 상태를 추적한다.

## 3. 사용자 문제

현재 재고관리 업무에서는 단순히 현재 수량만 보는 것으로는 충분하지 않습니다. 현업에서는 다음 질문에 빠르게 답할 수 있어야 합니다.

- 이 품목은 현재 몇 개 남아 있는가?
- 이 품목은 어느 창고, 어느 랙, 어느 셀에 있는가?
- 안전재고보다 부족한가?
- 입고 예정 또는 출고 대기 물량이 있는가?
- 이 품목은 어떤 군에 속하고 어떤 기준으로 관리해야 하는가?
- 최근 수량이 왜 변경되었는가?
- 담당자는 누구인가?

## 4. 제품 목표

- 실시간 재고 현황을 한눈에 확인할 수 있는 대시보드를 제공한다.
- 품목을 군별로 나누어 관리하고 필터링할 수 있게 한다.
- 품목 상세 화면에서 현업에 필요한 핵심 정보를 제공한다.
- 부족 재고, 입고 예정, 출고 대기, 재주문 권장 수량을 보여준다.
- 재고 변경 이력을 통해 수량 변동의 근거를 추적할 수 있게 한다.

## 7. 주요 기능 요구사항

### 재고 현황 조회

- 사용자는 전체 재고 현황을 실시간으로 확인할 수 있다.
- 사용자는 창고, 군, 상태별로 재고를 필터링할 수 있다.
- 사용자는 재고명 또는 품목번호로 검색할 수 있다.

### 자동 새로고침

- 시스템은 일정 주기마다 재고 데이터를 자동 갱신한다.
- 화면에는 자동 새로고침 상태와 최근 업데이트 시각을 표시한다.
- 사용자는 수동 새로고침도 실행할 수 있다.

### 군별 품목 관리

- 품목은 군별로 분류된다.
- 사용자는 전체, 안전용품, 포장자재, 전자부품, 운송자재 등 군별 탭을 통해 목록을 확인할 수 있다.
- 군별 재고 수량과 부족 재고 수를 확인할 수 있다.

### 품목 상세 조회

- 사용자는 재고 목록에서 품목을 선택해 상세 정보를 확인할 수 있다.
- 상세 화면에는 품목 사진, 무게, 크기, 군, 위치, 수량, 바코드, QR 코드가 포함된다.
- 상세 화면에는 안전재고, 최소 수량, 입고 예정, 출고 대기 정보가 포함된다.

### 부족 재고 알림

- 현재 수량이 안전재고 또는 최소 수량보다 낮으면 부족 상태로 표시한다.
- 시스템은 재주문 권장 수량을 표시한다.
- 부족 재고 알림 목록에서 품목 상세로 이동할 수 있다.

### 변경 이력

- 입고, 출고, 위치 이동, 재고 조정 이력을 기록한다.
- 이력에는 일시, 구분, 수량 변화, 변경 후 수량, 위치, 담당자, 비고를 포함한다.

## 9. MVP 범위

### 포함

- 실시간 재고 현황 대시보드
- 군별 필터
- 재고 검색
- 재고 목록 테이블
- 품목 상세 화면
- 부족 재고 알림
- 자동 새로고침 상태 표시
- 최근 변경 이력 표시

### 제외 또는 후순위

- 실제 ERP 연동
- 바코드 스캐너 하드웨어 연동
- 발주 자동 생성
- 다중 창고 간 복잡한 재고 이동 승인
- 회계 시스템 연동
- 모바일 앱

## 11. 성공 기준

- 창고관리자가 재고 부족 품목을 10초 이내에 찾을 수 있다.
- 재고관리자가 특정 품목의 위치, 수량, 안전재고, 입출고 상태를 한 화면에서 확인할 수 있다.
- 품목 상세 화면에서 크기, 무게, 군, 보관 조건, 담당자 정보를 확인할 수 있다.
- 최근 변경 이력을 통해 수량 변동의 이유를 추적할 수 있다.
- GitHub 이슈 또는 기획 회의에서 화면 예시를 기반으로 구현 범위를 논의할 수 있다.

## 12. 추가 논의가 필요한 질문

- 실제 관리할 품목군은 무엇인가?
- 품목 사진은 사용자가 직접 등록하는가, 외부 상품 DB에서 가져오는가?
- 바코드/QR 코드는 시스템에서 생성하는가, 기존 코드를 등록하는가?
- 재고 수량은 어떤 외부 시스템과 동기화되는가?
- 자동 새로고침 주기는 몇 초가 적절한가?
- 안전재고와 최소 수량은 품목별 수동 설정인가, 규칙 기반 자동 계산인가?
- 입고 예정과 출고 대기 데이터는 어디에서 가져오는가?
- 담당자와 권한 체계는 어떻게 나눌 것인가?
